# 简陋登录和聊天记录保存：从文件系统迁移到正规数据库的流程指南

## 1. 当前实现逻辑概述 (Current Implementation)

目前系统使用 **简单文件系统 (File-based JSON DB)** 实现持久化，适合原型快速验证和单机调试。

### 1.1 用户存储 (`data/users.json`)
- **格式**: JSON 数组。
- **字段**: `id` (注册时间戳), `username` (邮箱/用户名), `password` (明文)。
- **逻辑**: 登录时读取全量文件，通过 `Array.find` 验证。

### 1.2 聊天记录存储 (`data/chats/`)
- **命名规范**: `${username}_${characterId}.json` (例如: `xubing@gmail.com_fuxi.json`)。
- **文件结构**: 
  ```json
  {
    "userId": "xubing@gmail.com",
    "characterId": "fuxi",
    "messages": [
      { "role": "user", "content": "你好", "id": "1771267727846" },
      { "role": "assistant", "content": "你好...", "id": "1771267727847" }
    ]
  }
  ```
- **读写逻辑**: 
  - **读取**: 拼接路径，检查文件是否存在，解析 JSON 并返回 `messages` 数组。兼容纯数组格式。
  - **写入**: 每个新消息产生时，读取旧 JSON，`push` 新消息，全量覆盖写入文件。

---

## 2. 存在的局限性 (Limitations)

1. **性能瓶颈**: 随着聊天记录变长，每次发送消息都要读写整个 JSON 文件（I/O 压力大）。
2. **并发冲突**: 若同一用户在两个窗口同时发消息，可能导致文件覆盖丢失。
3. **扩展性**: 难以进行复杂查询（如：“搜索所有包含“八卦”的聊天记录”）。
4. **安全性**: 明文存储密码，且文件系统权限控制较弱。

---

## 3. 迁移到正规数据库的建议路线 (Migration Path)

建议选择 **PostgreSQL (推荐使用 Prisma ORM)** 或 **MongoDB**。

### 3.1 步骤一：定义数据库模型 (Schema Design)

**用户表 (Users):**
- `id` (UUID/Auto-increment)
- `email` (Unique)
- `password_hash` (使用 bcrypt 加密)
- `created_at`

**角色表 (Characters - 可选):**
- `id` (String)
  *目前在 `src/data/characters/` 静态存储，未来可落库以支持动态编辑。*

**聊天会话表 (Sessions/Threads):**
- `id` (UUID)
- `user_id` (Foreign Key)
- `character_id` (String)

**消息表 (Messages):**
- `id` (UUID)
- `session_id` (Foreign Key)
- `role` (Enum: user, assistant)
- `content` (Text)
- `created_at`

### 3.2 步骤二：数据搬迁脚本 (Data Migration)

编写一个 Node.js 脚本：
1. **迁移用户**: 读取 `users.json`，循环调用数据库 `insert`。
2. **迁移聊天**: 扫描 `data/chats/*.json`，解析文件，按照 `username` 关联用户 ID，再将 `messages` 插入消息表。

### 3.3 步骤三：更新后端逻辑 (`src/lib/db.ts`)

将 `fs.readFileSync/writeFileSync` 替换为数据库查询语句。
```typescript
// 改前 (current)
export const getChatHistory = (userId: string, characterId: string) => { 
  // ... fs.readFileSync ... 
}

// 改后 (future)
export const getChatHistory = async (userId: string, characterId: string) => {
  return await prisma.message.findMany({
    where: { userId, characterId },
    orderBy: { createdAt: 'asc' }
  });
}
```

---

## 4. 迁移时的主要注意事项

1. **ID 冲突**: 迁移时建议保留原始 `id` 作为参考，或全部替换为数据库生成的强唯一 ID (UUID)。
2. **格式一致性**: 确保 `role` 字段的值（user/assistant）在迁移前后保持一致，否则前端图标会显示错误。
3. **异步处理**: 数据库操作是异步的 (`async/await`)，需要更新所有调用链（API Routes 和组件内部调用）。
